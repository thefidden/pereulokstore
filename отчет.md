# Курсовая работа
## Разработка REST API интернет-магазина на Django + DRF

## 1. Техническое задание

### 1.1. Назначение системы

**Pereulok Store API** — REST API для интернет-магазина товаров для косплея и коллекционирования. Система позволяет управлять каталогом товаров, корзиной пользователей и заказами с интеграцией платёжной системы Альфа-Банка и авторизацией через Telegram.

### 1.2. Бизнес-логика

#### Основные бизнес-процессы:

**1. Управление каталогом товаров**
- Администраторы добавляют/редактируют/удаляют товары
- Товары классифицируются по типам: костюмы, фигурки, комиксы
- Каждый товар может иметь несколько изображений и тегов
- Товары с ценой < 0 или будущей датой поступления не отображаются в каталоге

**2. Работа с корзиной**
- Авторизованные пользователи добавляют товары в корзину
- Один товар может быть добавлен в корзину только один раз (UniqueConstraint)
- Пользователь может изменять количество товара в корзине

**3. Оформление заказов**
- Заказ создаётся из товаров, находящихся в корзине пользователя
- Цена товара фиксируется на момент создания заказа
- После создания заказа регистрируется платёж в Альфа-Банке
- Пользователь перенаправляется на страницу оплаты
- После успешной оплаты статус заказа меняется на "Собирается"

**4. Авторизация через Telegram**
- Пользователь сканирует QR-код или переходит по ссылке
- Telegram бот отправляет данные пользователя на сервер
- Создаётся сессия Django с привязкой к Telegram-аккаунту

#### Правила бизнеса:

1. **Цена товара**: может быть положительной (обычный товар) или равна -1 ("цена по запросу")
2. **Доступность товара**: товар доступен для покупки, если цена >= 0 и дата поступления <= текущей дате
3. **Уникальность в корзине**: один пользователь не может добавить один и тот же товар дважды
4. **Фиксация цены**: цена товара в заказе фиксируется на момент оформления и не изменяется при изменении цены товара

### 1.3. Роли пользователей

#### Роль: Администратор

**Назначение**: Управление каталогом товаров и заказами.

**Возможности**:

| № | Функция | Описание | API endpoint | HTTP метод |
|---|---------|----------|--------------|------------|
| 1 | Создание товара | Добавление нового товара с изображениями | `/api/products/` | POST |
| 2 | Редактирование товара | Изменение названия, цены, описания, категории | `/api/products/{id}/` | PATCH |
| 3 | Удаление товара | Удаление товара из каталога (PROTECT если в заказе) | `/api/products/{id}/` | DELETE |
| 4 | Просмотр всех заказов | Доступ к административной панели Django | `/admin/` | GET |
| 5 | Генерация отчётов | Создание PDF-отчёта по выбранным товарам | Admin action | - |

**Права**:
- Полный доступ к Django Admin (`is_staff=True`, `is_superuser=True`)
- CRUD операции над товарами (`IsAdminUser` permission)
- Просмотр и управление всеми заказами

#### Роль: Авторизованный пользователь (Customer)

**Назначение**: Покупка товаров и управление заказами.

**Возможности**:

| № | Функция | Описание | API endpoint | HTTP метод |
|---|---------|----------|--------------|------------|
| 1 | Просмотр каталога | Получение списка товаров с фильтрацией и поиском | `/api/products/` | GET |
| 2 | Добавление товара в корзину | Добавление товара в корзину с указанием количества | `/api/carts/` | POST |
| 3 | Изменение количества | Изменение количества товара в корзине | `/api/carts/{id}/` | PATCH |
| 4 | Удаление из корзины | Удаление товара из корзины | `/api/carts/{id}/` | DELETE |
| 5 | Оформление заказа | Создание заказа из товаров в корзине | `/api/orders/` | POST |
| 6 | Просмотр своих заказов | Получение списка заказов текущего пользователя | `/api/orders/` | GET |
| 7 | Проверка статуса оплаты | Проверка статуса платежа в Альфа-Банке | `/api/orders/{id}/payment/check/` | GET |
| 8 | Выход из системы | Деавторизация и удаление сессии | `/api/user/deauthenticate/` | GET |

**Права**:
- Просмотр каталога (`AllowAny`)
- Управление своей корзиной (`IsAuthenticated`)
- Управление своими заказами (`IsAuthenticated` + проверка владельца)

#### Роль: Гость (Anonymous User)

**Назначение**: Ознакомление с каталогом без авторизации.

**Возможности**:

| № | Функция | Описание | API endpoint | HTTP метод |
|---|---------|----------|--------------|------------|
| 1 | Просмотр каталога | Получение списка товаров с фильтрацией | `/api/products/` | GET |
| 2 | Просмотр информации о товаре | Получение детальной информации о конкретном товаре | `/api/products/{id}/` | GET |
| 3 | Инициация авторизации | Создание токена для авторизации через Telegram | `/api/auth-token/` | POST |

**Права**:
- Только чтение каталога (`AllowAny`)
- Невозможность добавления в корзину, оформления заказов

## 2. Валидация полей

### 2.1. Валидация на уровне модели (validators)

Для обеспечения целостности данных используются встроенные валидаторы Django:

```python
# server/app/api/models.py

from django.core.validators import MinValueValidator, MaxValueValidator

class Product(models.Model):
    price = models.IntegerField(
        validators=[
            MinValueValidator(0, message="Цена не может быть отрицательной"),
            MaxValueValidator(100_000, message="Цена не может превышать 100 000")
        ]
    )

class OrderProduct(models.Model):
    price = models.IntegerField(
        validators=[
            MinValueValidator(0),
            MaxValueValidator(100_000)
        ]
    )
    amount = models.IntegerField(
        validators=[
            MinValueValidator(1, message="Минимальное количество - 1"),
            MaxValueValidator(10, message="Максимальное количество - 10")
        ]
    )
```

**Описание валидаторов**:

| Поле | Валидатор | Значение | Сообщение об ошибке |
|------|-----------|----------|---------------------|
| `Product.price` | `MinValueValidator(0)` | >= 0 | Цена не может быть отрицательной |
| `Product.price` | `MaxValueValidator(100_000)` | <= 100000 | Цена не может превышать 100 000 |
| `OrderProduct.amount` | `MinValueValidator(1)` | >= 1 | Минимальное количество - 1 |
| `OrderProduct.amount` | `MaxValueValidator(10)` | <= 10 | Максимальное количество - 10 |

### 2.2. Валидация на уровне базы данных (Constraints)

Для обеспечения целостности данных на уровне БД используются constraints:

```python
# server/app/api/models.py

class Cart(models.Model):
    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=['user', 'product'],
                name='unique_user_product',
                violation_error_message='Этот товар уже добавлен в корзину'
            )
        ]
```

**Назначение**: Гарантирует, что пользователь не может добавить один и тот же товар в корзину дважды. При попытке создать дубликат выбрасывается `IntegrityError`.

### 2.3. Валидация на уровне сериализатора

Сериализаторы DRF автоматически применяют валидаторы модели при вызове `is_valid()`. Дополнительная валидация выполняется через поля сериализатора:

```python
# server/app/api/serializer.py

class ProductSerializer(serializers.ModelSerializer):
    name = serializers.CharField(
        max_length=50,
        error_messages={
            'max_length': 'Название не может превышать 50 символов'
        }
    )
    
    uploaded_images = serializers.ListField(
        child=serializers.ImageField(
            allow_empty_file=False,
            use_url=False
        ),
        write_only=True,
        required=False
    )
```

## 3. Оптимизация запросов

### 3.1. Проблема N+1 запросов

При итерации по связанным объектам Django выполняет отдельный запрос для каждого объекта. Например:

```python
# ❌ N+1 problem — 1 + N запросов
orders = Order.objects.all()
for order in orders:
    print(order.user.username)  # N запросов к User
    for product in order.products.all():  # N запросов к OrderProduct
        print(product.product.name)  # N запросов к Product
```

Для 10 заказов с 5 товарами в каждом = 1 + 10 + 50 + 250 = 311 запросов.

### 3.2. Решение: select_related и prefetch_related

#### select_related — для ForeignKey и OneToOne

Выполняет `JOIN` и загружает связанные объекты в одном запросе.

**Используется для**: ForeignKey, OneToOneField (прямые связи).

**Пример из проекта**:

```python
# server/app/api/models.py

class CartQuerySet(models.QuerySet):
    def optimized(self):
        """Оптимизирует запрос через select_related для ForeignKey связей.
        
        select_related используется для user и product, что выполняет
        JOIN и загружает связанные объекты в одном запросе.
        
        Returns:
            CartQuerySet: Оптимизированный QuerySet.
        
        Note:
            select_related используется для ForeignKey и OneToOne связей:
            - user: ForeignKey к User
            - product: ForeignKey к Product
            
            Это означает 1 запрос к БД с JOIN вместо N+1 запросов:
            SELECT cart.*, user.*, product.*
            FROM cart
            JOIN user ON cart.user_id = user.id
            JOIN product ON cart.product_id = product.id
        """
        return self.select_related('user', 'product')
```

**Результирующий SQL**:
```sql
SELECT cart.*, user.*, product.*
FROM cart
INNER JOIN user ON cart.user_id = user.id
INNER JOIN product ON cart.product_id = product.id
```

#### prefetch_related — для M2M и reverse FK

Выполняет отдельные запросы и объединяет в Python.

**Используется для**: ManyToManyField, reverse ForeignKey (related_name).

**Пример из проекта**:

```python
# server/app/api/models.py

class ProductQuerySet(models.QuerySet):
    def optimized(self):
        """Оптимизирует запрос через prefetch_related для связанных данных.
        
        Загружает связанные images и tags в отдельных запросах, избегая N+1.
        
        Note:
            prefetch_related используется для:
            - M2M связей (tags через ProductTag)
            - reverse FK связей (images — related_name от ProductImage)
            
            Это означает 3 запроса к БД вместо N+1:
            1. SELECT * FROM products
            2. SELECT * FROM product_images WHERE product_id IN (...)
            3. SELECT * FROM product_tags WHERE product_id IN (...)
        """
        return self.prefetch_related('images', 'tags')
```

**Результирующие SQL-запросы**:
```sql
-- Запрос 1: товары
SELECT * FROM product;

-- Запрос 2: изображения
SELECT * FROM product_image WHERE product_id IN (1, 2, 3, ...);

-- Запрос 3: теги
SELECT * FROM product_tag 
INNER JOIN tag ON product_tag.tag_id = tag.id
WHERE product_id IN (1, 2, 3, ...);
```

### 3.3. Комбинирование select_related и prefetch_related

Для сложных запросов используется комбинация обоих методов:

```python
# server/app/api/models.py

class OrderQuerySet(models.QuerySet):
    def optimized(self):
        """Оптимизирует запрос через комбинацию select_related и prefetch_related.
        
        select_related используется для:
        - user: ForeignKey к User (JOIN в одном запросе)
        
        prefetch_related используется для:
        - products: reverse FK к OrderProduct
        - products__product: ForeignKey к Product через OrderProduct
        - products__product__images: nested prefetch для изображений
        - products__product__tags: nested prefetch для тегов
        
        Returns:
            OrderQuerySet: Оптимизированный QuerySet.
        
        Note:
            Комбинация select_related + prefetch_related означает ~5 запросов:
            1. SELECT order.*, user.* FROM order JOIN user ...
            2. SELECT order_product.* FROM order_product WHERE order_id IN (...)
            3. SELECT product.* FROM product WHERE id IN (...)
            4. SELECT product_image.* FROM product_image WHERE product_id IN (...)
            5. SELECT product_tag.* FROM product_tag WHERE product_id IN (...)
            
            Без оптимизации было бы N+1 запросов для каждого уровня вложенности.
        """
        return self.select_related('user').prefetch_related(
            'products__product__images',
            'products__product__tags'
        )
```

### 3.4. Использование в views

Оптимизированные QuerySet используются в ViewSet:

```python
# server/app/api/views.py

class Products(viewsets.ViewSet):
    @action(methods=['get'], detail=False)
    def list(self, request: Request) -> Response:
        # Использование оптимизированного QuerySet
        products = Product.objects.all().optimized()
        # ...

class Carts(viewsets.ViewSet):
    @action(methods=['get'], detail=False, permission_classes=[IsAuthenticated])
    def list(self, request: Request) -> Response:
        # Использование оптимизированного QuerySet с аннотацией
        cart_items = Cart.objects.user_items(user).include_total_price().optimized()
        # ...

class Orders(viewsets.ViewSet):
    @action(methods=['get'], detail=False, permission_classes=[IsAuthenticated])
    def list(self, request: Request) -> Response:
        # Использование оптимизированного QuerySet
        orders = Order.objects.user_orders(user).optimized()
        # ...
```

### 3.5. Таблица сравнения методов оптимизации

| QuerySet | Метод | Тип оптимизации | Связи | SQL-запросов |
|----------|-------|-----------------|-------|--------------|
| `ProductQuerySet.optimized()` | `prefetch_related('images', 'tags')` | M2M и reverse FK | images — reverse FK, tags — M2M | 3 вместо N+1 |
| `CartQuerySet.optimized()` | `select_related('user', 'product')` | ForeignKey | user, product | 1 вместо 2N+1 |
| `OrderQuerySet.optimized()` | Комбинированный | FK + nested reverse FK | user (FK), products (reverse FK) | ~5 вместо N+1³ |

## 4. Сериализаторы

### 4.1. Назначение сериализаторов

Сериализаторы в DRF выполняют две функции:
1. **Сериализация** — преобразование complex types (QuerySet, Model) в JSON
2. **Десериализация** — валидация и преобразование JSON в Python-объекты

### 4.2. SerializerMethodField — вычисляемые поля

Используется для полей, которые отсутствуют в модели и требуют вычисления.

**Пример из проекта** — вычисление URL изображений:

```python
# server/app/api/serializer.py

class ProductSerializer(serializers.ModelSerializer):
    """Сериализатор для модели Product.
    
    Включает вычисляемые поля images и link.
    """
    
    images = serializers.SerializerMethodField()
    link = serializers.SerializerMethodField()
    
    class Meta:
        model = Product
        fields = ['id', 'type', 'name', 'price', 'description', 'images', 'link']
    
    def get_images(self, obj):
        """Возвращает список URL изображений товара.
        
        Args:
            obj (Product): Экземпляр товара.
            
        Returns:
            list[str]: Список относительных URL изображений.
        
        Note:
            Возвращает относительные URL. Для абсолютных URL
            передайте request в context сериализатора.
        """
        return [image.image.url for image in obj.images.all()]
    
    def get_link(self, obj):
        """Возвращает URL детальной страницы товара.
        
        Args:
            obj (Product): Экземпляр товара.
            
        Returns:
            str: URL вида /api/products/{id}/.
        """
        return obj.get_absolute_url()
```

### 4.3. Передача контекста в сериализатор

Контекст позволяет передавать дополнительные данные (например, request) в сериализатор для использования в SerializerMethodField.

**Пример использования request для построения абсолютных URL**:

```python
# В сериализаторе
class ProductSerializer(serializers.ModelSerializer):
    images = serializers.SerializerMethodField()
    
    def get_images(self, obj):
        """Возвращает абсолютные URL изображений.
        
        Использует request из контекста для build_absolute_uri.
        """
        request = self.context.get('request')
        if not request:
            return []
        
        return [
            request.build_absolute_uri(image.image.url)
            for image in obj.images.all()
        ]

# В view
class ProductViewSet(viewsets.ModelViewSet):
    def retrieve(self, request, pk):
        product = self.get_object()
        # Передача request в контекст
        serializer = ProductSerializer(product, context={'request': request})
        return Response(serializer.data)
```

### 4.4. Вложенные сериализаторы для чтения и записи

Для разных операций используются разные поля:

```python
# server/app/api/serializer.py

class CartSerializer(serializers.ModelSerializer):
    """Сериализатор для модели Cart.
    
    Включает вложенный ProductSerializer для чтения и product_id для записи.
    """
    
    # Для чтения — полный сериализатор с данными товара
    product = ProductSerializer(read_only=True)
    
    # Для записи — только ID товара
    product_id = serializers.PrimaryKeyRelatedField(
        queryset=Product.objects.all(),
        source='product',
        write_only=True
    )
    
    # Вычисляемое поле (аннотация на уровне QuerySet)
    total_price = serializers.IntegerField(read_only=True)
    
    class Meta:
        model = Cart
        fields = ['id', 'user', 'product', 'product_id', 'amount', 'total_price']
```

**Объяснение**:
- `product` (read_only=True) — используется при GET запросах, возвращает полные данные товара
- `product_id` (write_only=True) — используется при POST запросах, принимает только ID товара
- `source='product'` — указывает, что значение product_id записывается в поле product

### 4.5. Кастомный метод create

Для сложной логики создания объектов с вложенными данными:

```python
# server/app/api/serializer.py

class OrderSerializer(serializers.ModelSerializer):
    """Сериализатор для модели Order.
    
    Использует разные поля для чтения (products) и записи (added_products).
    """
    
    # Для чтения — позиции заказа
    products = OrderProductSerializer(many=True, read_only=True)
    
    # Для записи — данные для создания позиций
    added_products = OrderProductSerializer(many=True, write_only=True)
    
    class Meta:
        model = Order
        fields = ['id', 'user', 'price', 'status', 'products', 'added_products']
        read_only_fields = ['price']
    
    def create(self, validated_data):
        """Создаёт заказ с позициями из корзины.
        
        Процесс:
        1. Создаёт Order для пользователя
        2. Создаёт OrderProduct для каждой позиции
        3. Вычисляет общую сумму заказа
        
        Args:
            validated_data (dict): Данные с user и added_products.
            
        Returns:
            Order: Созданный заказ.
        
        Side Effects:
            - Создаёт Order запись
            - Создаёт OrderProduct записи с фиксацией цены
        """
        user = validated_data.pop('user')
        added_products = validated_data.pop('added_products')
        
        order = Order.objects.create(user=user)
        
        for added_product in added_products:
            product, amount = added_product['product'], added_product['amount']
            
            OrderProduct.objects.create(
                order=order,
                product=product,
                price=product.price,  # Фиксация цены на момент заказа
                amount=amount
            )
            
            order.price += product.price * amount
        
        order.save()
        return order
```

## 5. Пример использования аннотаций

### 5.1. Аннотация вычисляемых полей

Аннотации позволяют вычислять значения на уровне БД без N+1 запросов.

**Пример 1: Количество изображений товара**

```python
# server/app/api/models.py

class ProductQuerySet(models.QuerySet):
    def with_images_count(self):
        """Добавляет annotated-поле images_count к каждому товару.
        
        Полезно для отображения количества изображений без N+1 запросов.
        
        Returns:
            ProductQuerySet: QuerySet с аннотированным полем images_count.
        
        Example:
            >>> products = Product.objects.all().with_images_count()
            >>> products[0].images_count
            3
        """
```

### 6.2. Создание FilterSet

```python
# server/app/api/filters.py

"""Фильтры для API интернет-магазина.

Содержит django_filters.FilterSet для фильтрации товаров.
Поддерживает:
- Фильтрацию по категории
- Диапазон цен
- Триграммный поиск по названию
- Фильтрацию по тегам
"""

import django_filters
from django.db.models import Q
from django.contrib.postgres.search import TrigramSimilarity
from .models import Product


class ProductFilter(django_filters.FilterSet):
    """Фильтр для модели Product.
    
    Предоставляет фильтры:
    - type: Точное совпадение категории (case-insensitive)
    - price_min: Минимальная цена (gte)
    - price_max: Максимальная цена (lt)
    - name: Поиск по названию (триграммный + contains)
    - tag: Фильтр по slug тега
    
    Example:
        GET /api/products/?type=suit&price_min=1000&price_max=5000
        GET /api/products/?name=iron&tag=new
    """
    
    type = django_filters.ChoiceFilter(
        field_name='type',
        choices=Product.PRODUCT_TYPES,
        lookup_expr='iexact'
    )
    
    price_min = django_filters.NumberFilter(
        field_name='price',
        lookup_expr='gte'
    )
    price_max = django_filters.NumberFilter(
        field_name='price',
        lookup_expr='lt'
    )
    
    created_after = django_filters.IsoDateTimeFilter(
        field_name='creation_date',
        lookup_expr='gte'
    )
    created_before = django_filters.IsoDateTimeFilter(
        field_name='creation_date',
        lookup_expr='lte'
    )
    
    name = django_filters.CharFilter(method='filter_by_name')
    
    tag = django_filters.CharFilter(
        field_name='producttag__tag__slug',
        lookup_expr='iexact'
    )
    
    def filter_by_name(self, queryset, name, value):
        """Фильтрует товары по названию с использованием триграммного поиска.
        
        Комбинирует два подхода:
        1. name__icontains — точное вхождение подстроки
        2. TrigramSimilarity — поиск похожих слов (для опечаток)
        
        Args:
            queryset: Исходный QuerySet.
            name: Имя фильтра (игнорируется).
            value (str): Строка для поиска.
        
        Returns:
            QuerySet: Отфильтрованный и отсортированный по релевантности QuerySet.
        
        Note:
            Триграммный поиск работает только с PostgreSQL и расширением pg_trgm.
            Порог similarity > 0.2 можно настроить под конкретные требования.
        """
        if not value:
            return queryset
        
        return (
            queryset
            .annotate(similarity=TrigramSimilarity('name', value))
            .filter(Q(name__icontains=value) | Q(similarity__gt=0.2))
            .order_by('-similarity', 'name')
        )
    
    class Meta:
        model = Product
        fields = ['type', 'price_min', 'price_max', 'name', 'tag']
```

### 6.3. Использование в ViewSet

```python
# server/app/api/views.py

from .filters import ProductFilter

class Products(viewsets.ViewSet):
    """ViewSet для CRUD операций с товарами."""
    
    serializer_class = ProductSerializer
    filter_backends = [DjangoFilterBackend]
    filterset_class = ProductFilter
    
    @action(methods=['get'], detail=False)
    def list(self, request: Request) -> Response:
        """Возвращает список товаров с возможностью фильтрации.
        
        Query Params:
            type: Категория товара (suit, shape, comics).
            price_min: Минимальная цена.
            price_max: Максимальная цена.
            name: Подстрока для поиска в названии.
            tag: Slug тега для фильтрации.
        """
        products = Product.objects.all().optimized()
        filterset = ProductFilter(request.query_params, queryset=products)
        
        if not filterset.is_valid():
            return Response(filterset.errors, status=status.HTTP_400_BAD_REQUEST)
        
        products_filtered = filterset.qs
        serializer = ProductSerializer(products_filtered, many=True)
        return Response(serializer.data, status=status.HTTP_200_OK)
```

```python
# server/app/api/utils.py

def register_order(order: 'Order') -> str:
    """Регистрирует заказ в платёжной системе Альфа-Банка.

    Отправляет запрос к API Альфа-Банка для создания платежа.
    Возвращает URL страницы оплаты.

    Args:
        order (Order): Заказ для регистрации.

    Returns:
        str: URL страницы оплаты (formUrl).

    Raises:
        requests.RequestException: При ошибке сети или таймауте.
        KeyError: Если ответ банка не содержит formUrl.

    Environment:
        ALFA_BANK_TOKEN: API токен из settings.
        ALFA_BANK_URL: URL API (захардкожен как https://alfa.rbsuat.com).

    Note:
        Используется тестовый URL Альфа-Банка (rbsuat.com).
        Для продакшена замените на production URL.

    Example:
        >>> order = Order.objects.get(id='...')
        >>> payment_url = register_order(order)
        >>> # Redirect user to payment_url
        'https://alfa.rbsuat.com/payment/...'
    """
    request = requests.post(
        url='https://alfa.rbsuat.com/payment/rest/register.do',
        data={
            'token': settings.ALFA_BANK_TOKEN,
            'orderNumber': order.id,
            'amount': order.price,
            'clientId': order.user.id,
            'returnUrl': f'https://pereulokstore.ru/orders/{order.id}/payment/check'
        }
    )
    response = request.json()
    return response.get('formUrl')
```

## 10. Sentry — мониторинг ошибок

### 10.1. Что такое Sentry

**Sentry** — платформа для мониторинга ошибок в реальном времени. Автоматически отлавливает исключения, логирует их и отправляет уведомления разработчикам.

**Основные возможности**:
- Автоматический захват необработанных исключений
- Группировка похожих ошибок
- Контекст ошибки (пользователь, request, окружение)
- Интеграция с Slack, Email, Telegram
- Release tracking
- Performance monitoring
- Source maps для JavaScript

### 10.2. Подключение Sentry

**Установка**:
```bash
pip install sentry-sdk
```

**Настройка в settings.py**:
```python
import sentry_sdk

# Инициализация только в продакшене
if not DEBUG:
    sentry_sdk.init(
        dsn='https://xxx@sentry.io/xxx',  # DSN из настроек проекта Sentry
        traces_sample_rate=1.0,  # 100% транзакций для мониторинга
        profiles_sample_rate=1.0,  # 100% профилирования
    )
```

### 10.3. Пример использования

**Автоматический захват исключений**:
```python
# Sentry автоматически захватывает все необработанные исключения
def some_view(request):
    result = 1 / 0  # ZeroDivisionError будет отправлена в Sentry
    return Response({'result': result})
```

**Ручная отправка исключений**:
```python
from sentry_sdk import capture_exception, capture_message

def register_order(order):
    """Регистрирует заказ в платёжной системе."""
    try:
        response = requests.post(payment_url, data=order_data, timeout=10)
        response.raise_for_status()
    except requests.RequestException as e:
        # Отправка ошибки в Sentry с контекстом
        capture_exception(e)
        raise PaymentError(f'Payment registration failed: {e}')

# Отправка сообщения (warning, info)
capture_message('Payment system response time is high', level='warning')
```

**Добавление контекста**:
```python
import sentry_sdk

def process_payment(order):
    try:
        # Бизнес-логика
        pass
    except Exception as e:
        # Добавление контекста перед отправкой
        with sentry_sdk.push_scope() as scope:
            scope.set_tag('payment', 'failed')
            scope.set_user({'id': order.user.id, 'username': order.user.username})
            scope.set_context('order', {
                'id': str(order.id),
                'price': order.price,
                'status': order.status
            })
            scope.set_extra('bank_response', response.text)
            capture_exception(e)
        raise
```

## 11. Django Silk — профилирование запросов

### 11.1. Что такое Django Silk

**Django Silk** — инструмент для профилирования Django-приложений. Позволяет анализировать:

- SQL-запросы и их время выполнения
- HTTP-запросы и ответы
- Использование памяти
- N+1 проблемы
- Дублирующиеся запросы

### 11.2. Подключение Django Silk

**Установка**:

**Настройка в settings.py**:

### 11.4. Анализ результатов

Интерфейс Django Silk показывает:

**Requests** — список всех HTTP-запросов:
- URL, метод, статус
- Время выполнения
- Количество SQL-запросов
- Время SQL-запросов

**SQL Queries** — детализация SQL-запросов:
- Текст запроса
- Время выполнения
- План запроса (EXPLAIN)
- Трассировка (откуда вызван)

**Profiles** — профили функций:
- Время выполнения
- Использование памяти

### 11.5. Выявление проблем

**Пример выявления N+1 проблемы**:

1. Открыть `/silk/requests/`
2. Найти запрос `/api/user/`
3. Посмотреть количество SQL-запросов (например, 15)
4. Открыть SQL Queries для этого запроса
5. Увидеть повторяющиеся запросы к таблицам `cart`, `order`, `product`
6. Добавить `select_related` / `prefetch_related`
7. Перезапросить и проверить, что запросов стало меньше

В текущем проекте Django Silk **не подключён**. Для подключения необходимо:

```python
# До оптимизации
# GET /api/user/ — 15 SQL запросов, 120ms

# После добавления prefetch_related
class UserSerializer(serializers.ModelSerializer):
    cart = CartSerializer(many=True, read_only=True)
    orders = OrderSerializer(many=True, read_only=True)
    
    @classmethod
    def get_optimized_queryset(cls, user):
        return User.objects.prefetch_related(
            'cart__product__images',
            'orders__products__product'
        ).get(pk=user.pk)

# GET /api/user/ — 5 SQL запросов, 45ms
```
